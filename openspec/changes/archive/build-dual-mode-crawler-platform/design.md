## Context

webmagic-ng 目标不是单一爬虫框架，而是一个覆盖“生成、运行、发布、调度、治理”的双模式爬虫平台。仓库目前几乎为空，适合先定义清楚核心对象模型和阶段边界，再开始编码。当前约束有三点：

1. Quick Start 模式必须让使用者以最低认知成本完成试抓和本地调试，否则后续 Server 模式不会有稳定的爬虫来源。
2. Server 模式不能依赖与本地模式完全不同的运行语义，否则发布链路会变成“重写一遍爬虫”。
3. 管理与观测能力必须在早期进入设计，而不是等执行链路完成后补救，否则调度、权限和日志模型会频繁返工。

涉及的主要角色包括：爬虫开发者、任务提交者、平台管理员。

## Goals / Non-Goals

**Goals:**

- 定义统一的爬虫项目模型，使本地开发产物可以直接发布到服务端运行。
- 建立 Quick Start MVP，覆盖自然语言到本地可执行爬虫的完整闭环。
- 建立 Server MVP，覆盖鉴权、任务提交、执行调度、定时任务和日志查询。
- 建立基础 Admin 观测台，支持管理员查看任务状态、日志和失败详情。
- 将整体建设拆分为可连续交付的 4 个阶段，每阶段都有明确可验收目标。

**Non-Goals:**

- 本阶段不追求分布式多地域、多租户计费和复杂资源配额系统。
- 本阶段不实现通用可视化爬虫编排器或拖拽式低代码设计器。
- 本阶段不承诺支持所有反爬场景，只提供标准浏览器自动化和可扩展执行框架。
- 本阶段不建设完整的 BI 报表系统，只提供任务与运行层面的观测面板。

## Decisions

### 1. 以“统一爬虫项目模型”作为双模式共享底座

爬虫项目应至少包含：目标站点元数据、输入参数定义、抓取流程代码、提取 schema、运行配置、依赖声明和发布清单。Quick Start 负责生成和调试该项目，Server 模式负责接收和执行同一产物。

原因：

- 避免本地模式和服务端模式使用两套 DSL 或两套 runtime。
- 后续可以用同一套打包格式支持 CLI 执行、容器执行和远端 Worker 执行。

备选方案：

- 方案 A：本地直接生成脚本，服务端单独定义任务配置。缺点是发布时必然要做二次适配。
- 方案 B：先只做 Server 任务模型，本地模式通过 API 调远端试跑。缺点是本地开发体验差，且早期依赖太多基础设施。

### 2. Quick Start 采用“两段式链路”：页面分析 Agent + 可审计代码固化

Quick Start 不直接执行一个黑盒 Agent，而是拆成：

- 页面分析阶段：调用浏览器自动化工具读取页面结构、定位候选字段、生成抓取建议。
- 代码固化阶段：将建议写入框架规定的项目文件和运行入口，用户可以查看、修改和再次执行。

原因：

- 可审计、可调试，失败时容易定位是页面理解错误还是运行逻辑错误。
- 更适合作为 Skill 提供给 Codex/Claude Code 等 Coding Agent。

备选方案：

- 直接让 Agent 持续控制浏览器完成抓取。缺点是稳定性差、结果难复现、难以发布到 Server。

### 3. Server MVP 采用控制面/执行面分离

建议至少拆分为以下模块：

- API Server：接收项目发布、任务创建、任务查询、日志查询。
- Scheduler：处理 Cron、重试和待执行任务入队。
- Worker：拉取任务、启动运行时、执行爬虫并上报状态。
- Metadata Store：保存项目版本、任务实例、运行状态、鉴权主体。
- Log Store：保存结构化日志和错误堆栈。

原因：

- 后续扩容主要发生在 Worker，不应让 API 和调度逻辑绑定在同一进程。
- 日志与元数据职责分开，更容易支持按任务和按用户隔离查询。

备选方案：

- 单体进程内同时做 API、Cron、执行。优点是实现快，但一旦执行阻塞或浏览器资源泄漏，会拖垮整个服务。

### 4. 鉴权以 API Key 为 MVP，权限模型分“两类主体”

MVP 只支持 API Key 鉴权，但从数据模型上区分：

- 提交者主体：只能管理自己提交的项目、任务、日志。
- 管理员主体：可查看全局任务、失败日志、节点状态和告警信息。

原因：

- 满足早期平台接入门槛低的要求。
- 为未来扩展到组织、角色和更细粒度 RBAC 预留空间。

备选方案：

- 直接接入完整 OAuth/RBAC。缺点是会拖慢 MVP。

### 5. Roadmap 采用“四阶段递进式交付”

阶段划分如下：

- Phase 0: Foundation
  - 明确 monorepo/module 边界、统一配置、日志协议、爬虫项目模型。
- Phase 1: Quick Start MVP
  - 打通自然语言到本地 CLI 可运行爬虫的闭环。
- Phase 2: Server MVP
  - 打通发布、任务提交、定时调度、Worker 执行、日志回查。
- Phase 3: Admin & Production Hardening
  - 补齐管理后台、监控告警、失败重试策略、部署文档和灰度方案。

原因：

- 先证明“能抓”，再证明“能稳定跑”，最后证明“能运维”。
- 每个阶段都能形成独立 demo 和验收标准，便于融资、试点或团队协同。

## Risks / Trade-offs

- [Quick Start 对 LLM/页面分析依赖较重] → 通过保留手工编辑入口、增加选择器回放验证和详细日志降低不可控性。
- [统一项目模型设计过早可能导致抽象失真] → 只覆盖本地与服务端都必须共享的最小字段，避免一开始做成庞大 DSL。
- [浏览器执行资源消耗高，Server MVP 容易不稳定] → Worker 采用独立进程/容器执行，控制并发、超时和回收策略。
- [Admin 需求容易膨胀成完整运维平台] → MVP 只做任务列表、实例详情、日志查询和失败筛选，不做复杂报表。
- [本地与服务端运行环境差异可能导致“本地可跑、线上失败”] → 定义统一运行镜像、依赖声明与环境检查命令。

## Migration Plan

由于当前仓库尚无历史实现，本次变更不涉及兼容旧系统迁移。部署推进顺序建议如下：

1. 先完成项目骨架与统一模型，确保 Quick Start 和 Server 共享核心包。
2. 交付 Quick Start MVP，使用本地 CLI 验证至少 2 类真实站点。
3. 在独立环境部署 Server MVP，验证项目发布、单次任务、Cron 任务和日志隔离。
4. 上线 Admin MVP 和基础监控后，再开放给更广泛使用者。

回滚策略：

- 各阶段均以模块化发布，CLI 和 Server 独立版本管理。
- Server 侧功能通过 feature flag 控制新调度器、新日志查询或新 Admin 页面开关。

## Open Questions

- Quick Start 生成代码时，首个版本采用哪种语言和运行时作为主路径。
- Worker 执行隔离是优先使用本地进程池、容器还是作业平台。
- 结果存储是否需要在 MVP 阶段就支持对象存储和结构化导出。
- Admin 管理端优先做 Web，还是同时维护桌面 Client 包装层。
